## 2. 推荐仓库结构


## 3. 开发环境

### 3.1 基础要求

| 项 | 建议版本 |
| --- | --- |
| Python | 3.11+ |
| CUDA（本地 GPU 训练/推理） | 与 PyTorch 官方矩阵一致 |
| Docker | 24+（用于一致化运行） |

### 3.2 核心 Python 依赖（示例）

- `langchain`、`langchain-community`、`langchain-openai`（或对应厂商集成包）
- `fastapi`、`uvicorn[standard]`
- `pydantic-settings`
- 向量库 SDK：`pymilvus` / `qdrant-client` / `pgvector` + SQLAlchemy
- 训练：`torch`、`transformers`、`peft`、`trl`、`datasets`、`accelerate`

**锁定版本**：生产使用 `requirements.lock` 或 `uv.lock` / `poetry.lock`，避免隐式升级导致行为变化。

### 3.3 本地启动（概念步骤）

1. 复制 `.env.example` 为 `.env`，填入模型 API Key、数据库连接等。
2. `docker compose up -d` 启动向量库、Redis、MinIO（若采用）。
3. `uvicorn src.api.main:app --reload` 启动 API。
4. 通过 Swagger `/docs` 调试上传与问答接口。

## 4. 配置管理

| 配置类 | 示例项 | 说明 |
| --- | --- | --- |
| 模型 | `BASE_LLM`、`EMBEDDING_MODEL`、`RERANKER_MODEL` | 支持多环境 profile |
| RAG | `CHUNK_SIZE`、`CHUNK_OVERLAP`、`TOP_K`、`USE_HYBRID`、`USE_RERANK` | 可运行时热更（二期）或重启加载 |
| 安全 | `JWT_SECRET`、`SESSION_TTL`、`AUDIT_SINK` | 禁止提交真实密钥到 Git |
| 微调 | `TRAIN_DATA_PATH`、`OUTPUT_DIR`、`LORA_R`、`LORA_ALPHA` | 训练机与环境变量 |

使用 `pydantic-settings` 统一校验缺失项，启动失败要 **快速显式报错**。

## 5. 开发流程与协作

### 5.1 分支策略（建议）

- `main`：可发布稳定分支，受保护。
- `develop`：日常集成（小团队可简化为仅 `main` + feature）。
- `feature/<ticket>-short-name`：功能分支，合并前通过 CI。
- `hotfix/*`：生产紧急修复。

### 5.2 代码规范

- Formatter：`ruff format` 或 `black`（团队择一）。
- Lint：`ruff`。
- 类型：`mypy` 可选，核心模块建议逐步加类型注解。

### 5.3 提交信息

使用完整句说明「做了什么、为何」，例如：`Add hybrid retrieval flag to settings for BM25 toggle`。

### 5.4 Code Review 关注点

- 权限是否在 **检索前** 与 **返回引用前** 双重校验。
- Prompt 是否注入不可信字符串（防注入模板）。
- 大文件与长耗时任务是否异步队列化，避免阻塞 worker。

## 6. LangChain 开发约定

- **链路与 Retriever** 放在 `src/rag/chains/`，对外只暴露应用层用例接口，便于单测 mock 向量库。
- 使用 **Callbacks** 或 LangSmith 记录 trace_id，与业务日志关联。
- 长文档分块策略需 **版本化**（变更 chunk 策略应 bump 索引版本）。
- 金融场景优先 **显式引用字段**（chunk_id、page）进入 prompt，便于后处理校验。

## 7. 数据与微调开发

### 7.1 指令数据格式（示例）

每行 JSON，字段与训练脚本对齐，例如：

```json
{
  "instruction": "根据给定材料回答问题。",
  "input": "材料：……\n问题：某公司 2024 年营收同比增长多少？",
  "output": "根据年报披露，同比增长为 X%。",
  "meta": {"source_doc_ids": ["doc-uuid-1"], "annotator": "internal-v1"}
}
```

### 7.2 训练流程 Checklist

- [ ] 数据已通过合规审核与去标识化（如需要）
- [ ] 训练/验证划分无文档级泄漏（同文不同问需同 split 策略文档化）
- [ ] 保存：`adapter`、tokenizer、`training_args`、数据版本 hash
- [ ] 在固定评测集上对比 **仅 RAG / RAG+基座 / RAG+LoRA**

### 7.3 模型上线 Checklist

- [ ] 推理镜像与依赖版本与评测环境一致
- [ ] LoRA 与基座版本组合在注册表可查
- [ ] 回滚方案：切换路由到上一 model_id
- [ ] 监控：错误率、延迟、token 用量、拒答率

## 9. CI/CD 建议

- **CI**：lint + unit tests + OpenAPI schema 校验；不默认跑 GPU 训练。
- **CD**：镜像构建推送到私有仓库；K8s 滚动更新；数据库迁移 job 与 API 版本协调。
- **密钥**：CI 使用 OIDC 或密钥管理，禁止长期明文 token。

## 11. 常见问题（FAQ）

**Q：先上 RAG 还是先微调？**  
A：通常先 RAG 与基座打通端到端，再用小规模高质量数据做 LoRA，增量迭代。

**Q：向量库选型犹豫？**  
A：小规模 PGVector 运维最轻；亿级且需复杂过滤可考虑 Milvus/Qdrant。

**Q：LangChain 版本升级？**  
A：固定 minor 版本，升级前跑黄金集与集成测试；关注 Retriever API 破坏性变更。
